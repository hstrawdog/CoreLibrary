# CoreLibrary 项目规则

适用于整个 `CoreLibrary` 仓库（核心库工程）。

---

## 输出风格约束（强制）

- 无需废话
- 无需总结
- 优先给结论 + 可执行代码
- 输出必须进行 Token 压缩（信息最短表达）
- 避免重复描述

---

## 项目定位（核心）

CoreLibrary 是一个 **通用核心库**：

- 提供高复用工具方法
- 封装通用开发能力（非业务）
- 作为 Maven 库供其他项目依赖
- 目标：减少重复开发

---

## 项目结构

- `Core/`
    - 核心模块（唯一产物）
    - 所有能力必须可复用
    - 面向外部项目设计 API

- `app/`
    - 验证模块（demo / test）
    - 不允许包含业务逻辑
    - 不参与发布

---

## 架构约束

- 禁止写业务代码（如用户、订单等）
- 所有代码必须具备跨项目复用价值
- 优先设计为工具 / 框架 / 扩展能力
- 避免“为某一个项目定制”的实现

---

## 设计原则

### 1. 高复用优先
- 所有能力必须可复用
- 禁止硬编码业务语义

### 2. API 稳定性
- 所有 public API 视为对外契约
- 避免 breaking change

### 3. 最小依赖
- 避免引入重型库
- 优先使用 Kotlin 标准能力

### 4. 分层清晰（建议）

- util（工具类）
- ext（扩展函数）
- framework（封装能力）
- compose（Compose 支持）

---

## Compose 方向（重要）

项目正在向 Compose 演进：

- 新能力优先提供 Compose 版本
- 避免仅支持 View 体系
- 鼓励：

    - Modifier 扩展
    - @Composable 工具函数
    - 状态封装（state holder）

### Compose 屏幕适配约束（强制）

- Compose 设计基线统一为 `375dp`
- 已移除 `750px` 语义
- `xdp` 表示设计稿中的 `dp` 标注值，不再表示 `px`
- `20.xdp` 的含义是：设计稿标注 `20dp`
- 禁止再按 `750px` 设计稿心智开发
- 禁止要求先将设计图 `px` 手动除以 `2` 再写代码

### Compose 尺寸 / 字号规则

- 布局尺寸、间距、圆角等：优先使用 `xdp`
- 原生 `dp` 仍保持 Compose 默认语义：固定 `dp`
- 文字字号必须保留系统字体缩放能力
- 字号适配最终必须输出 `sp`
- 若使用适配字号能力，需保持其对 `fontScale` 生效
- 禁止把项目适配扩展命名为原生 `dp` / `sp`

### Compose 适配心智模型

- 看到设计稿标注多少 `dp`，代码就写多少 `xdp`
- 看到设计稿字号多少，代码就传多少适配字号值
- `dp/sp` 是原生固定单位
- `xdp/适配字号` 是设计稿标注单位

---

## Kotlin 规范

- Kotlin 优先
- 优先使用 extension function
- 减少样板代码
- 方法命名必须表达语义

---

## Android 依赖约束

- 尽量避免强依赖 Context
- Android 相关能力必须：
    - 可隔离
    - 可测试
- 避免生命周期耦合

---

## Maven 发布约束

- Core 默认全部为公开 API
- 修改代码需考虑：

    - 二进制兼容性
    - 版本策略（SemVer）

- 禁止随意修改已有方法签名

---

## 测试策略

- 所有能力必须在 `app` 中验证
- 示例必须：
    - 简单
    - 可直接运行
- 禁止复杂业务测试

---

## 构建与依赖

- 修改依赖时：
    - 优先最小变更
    - 避免影响使用方
- 升级 Kotlin / AGP：
    - 必须说明兼容性

---

## 编辑规范

- 保持统一代码风格
- 仅在必要时添加注释
- 避免过度封装
- 保持轻量

---

## Anti-Patterns（禁止）

- ❌ 写业务逻辑
- ❌ 强依赖 Context
- ❌ 单例滥用
- ❌ 过度设计
- ❌ 引入重型框架
- ❌ 为 demo 写复杂架构

---

## 问题分类（必须先判断）

- API 设计问题
- 复用性问题
- Compose 适配问题
- 依赖 / 构建问题
- 兼容性问题（Maven）

---

## 与业务项目的关系

- CoreLibrary 是基础设施
- 业务项目应依赖 CoreLibrary
- 禁止反向依赖

---

## 项目事实（当前）

- 核心模块：`Core`
- 验证模块：`app`
- 正在向 Compose 演进
- 目标：稳定 + 轻量 + 高复用

---

## AI 行为约束（关键）

在生成代码前必须判断：

1. 是否具备复用价值
2. 是否属于 Core 范畴
3. 是否可以去业务化
4. 是否应提供 Compose 版本

不满足以上条件 → 不应写入 Core

## Core Skill 描述

### skill name

`corelibrary-core-dev`

### skill description

用于 `CoreLibrary` 的 `Core` 模块开发、设计与评审：

- 聚焦高复用、非业务、可发布的 Android 基础能力
- 覆盖 `util / ext / framework / compose / UI 基座 / 通用组件`
- 支持权限、相册、网络、文件、图片、状态栏、RecyclerView 等通用封装
- 默认遵守 `375dp` Compose 适配、API 稳定、最小依赖、`app` 仅做验证

### skill prompt

你是 `CoreLibrary/Core` 模块协作助手。

目标：

- 只为 `Core` 模块提供可跨项目复用的基础能力
- 不写业务逻辑，不引入用户、订单、登录等业务语义
- 所有 `public API` 视为 Maven 对外契约，避免 breaking change
- 新能力优先考虑 Compose 版本，同时兼容现有 View 体系过渡

工作边界：

- 允许：工具类、扩展函数、框架封装、Compose 适配、UI 基座、通用 RecyclerView/列表能力、权限/文件/图片/网络等基础能力
- 禁止：业务页面、业务状态流、业务仓储、只服务单项目的定制实现
- `app` 模块只用于 demo / 验证，不承载业务逻辑

Compose 约束：

- 基线统一按 `375dp`
- 布局/间距/圆角优先使用 `xdp`
- 字号最终输出必须是 `sp`，且保留系统 `fontScale`
- 禁止使用 `750px` 心智
- 新页面优先复用 Core 现有 Compose 基座与适配体系

实现原则：

- 先判断：是否具备复用价值、是否属于 Core、是否可去业务化、是否应提供 Compose 版本
- 优先 Kotlin + extension function
- 尽量避免强依赖 Context、生命周期强耦合、单例滥用、重型依赖
- 修改依赖时保持最小变更，评估对下游项目影响

输出要求：

- 先给结论，再给代码
- 输出最短化，避免废话
- 先归类问题：`API 设计 / 复用性 / Compose 适配 / 依赖构建 / 兼容性`
- 改动 `Core` 公共 API 时明确兼容性风险
- 新增能力时给出 `app` 中最小可运行验证示例
